Edit email client configuration
Use the Edit email client configuration page to view or specify the settings for the email client that will retrieve service requests that are sent to the Integration Server via email.
Note:
You cannot edit port settings for the primary port.
Click Return to Ports to return to the list of ports that are currently defined on the server.
Package
Package Name
Specifies the name of the package that you want to associate with this port. When you assign a package to a port, the server enables and disables the port in conjunction with the package--if you disable the port, the server disables the port automatically. A package that you assign to a port also retains its port assignment when it is published to other servers. If a subscriber installs the package, an Email client with these same settings is created automatically on the subscriber's server. If you do not have a package to assign to the port, leave the Package Name setting at WmRoot (the server's core package). This makes the port available whenever the server is running.
Alias
An alias for the port. An alias must include one or more of the following: ASCII characters, numbers, underscore (_), period (.), and hyphen (-).
Description
A description of the port.
Server Information
Use this section to specify information about the IMAP or POP3 server from which this email client will retrieve service requests.
Enable
Controls whether or not the port is enabled. When the port is enabled, it can receive requests.
Note:
A disabled port is not re-enabled by restarting the server. A disabled port does not accept requests until you explicitly enable it.
Type
Specifies the type of mail server that runs on Host Name. Select IMAP or POP3.
Host Name
Specifies the name of the machine on which the IMAP or POP3 mail server is running.
Port (optional)
Specifies the port on which the mail server listens for requests. (This is the port where your Integration Server will retrieve email messages.) The default port used by IMAP is 143; the default port used by POP3 is 110.
User Name
Specifies the user name that your server must use to log on to the mail server.
Authentication
Specifies the type of authentication that Integration Server uses to connect to the specified e-mail server. Select Basic Authentication or OAuth.
Note:
OAuth is not supported for resource e-mail notifications.
Password
Specifies the password for User Name. This field applies only to Basic Authentication.
Auth URL
Specifies the URL of the endpoint that Integration Server uses to request authorization code. This field applies only to OAuth.
Client ID
Specifies the unique public identifier that the OAuth server generates for Integration Server during registration. This field applies only to OAuth.
Client Secret
Specifies a unique string that the OAuth server provides to Integration Server. It is known only to Integration Server and the authorization server. This field applies only to OAuth.
Scope
Specifies the mail server access permissions configured for Integration Server during registration. You can specify multiple scopes separated by a space. This field applies only to OAuth.
Access Token URL
Specifies the URL that Integration Server uses to request access token from the OAuth server. This field applies only to OAuth.
Redirect URL
Specifies the URL that the OAuth server uses to send authentication responses to Integration Server.
If Integration Server is accessed locally, enter the redirect URLs in one of the applicable formats:
http://localhost:{port}/WmRoot/security-oauth-get-authcode.dsp or https://localhost:{port}/WmRoot/security-oauth-get-authcode.dsp
If Integration Server is accessed remotely, use https://{ISHostName}:{port}/WmRoot/security-oauth-get-authcode.dsp.
This field applies only to OAuth.
Authorization Code
Click Get Authorization Code to receive a unique string from the OAuth Server that Integration Server requires to request an access token.
Note:
Whenever you edit the OAuth credentials for an email port, ensure to get the authorization code.
Access Token Expiry Time
Specifies the time until when an access token issued to Integration Server is valid. This field applies only to OAuth. This is a read-only field and is generated based on the access token.
Transport Layer Security
Specifies the type of SSL encryption that Integration Server uses when communicating with an e-mail server. You can configure the port to use one of the following options:
Option
Description
None
Default. Use a non-secure mode when communicating with an e-mail server. When you specify None, the e-mail server authenticates the e-mail client using only the username and password.
Explicit
Use explicit security when communicating with an e-mail server.With explicit security, Integration Server establishes an un-encrypted connection to the e-mail server and then upgrades to the secure mode. With explicit Transport layer Security, Integration Server can communicate with e-mail servers that support and do not support SSL encryption. If the email server does not support Transport Layer Security, Integration Server will disconnect the connection established to the e-mail server. You can then establish an un-secure connection with the e-mail server by selecting the None option in the Transport Layer Security field and enabling the port.
Implicit
Use implicit security when communicating with an e-mail server. With implicit security, Integration Server always establishes an encrypted connection to the e-mail server. Only clients that support SSL will be permitted access.
Truststore Alias (optional)
Optional. Alias for the truststore that contains certificates presented by the e-mail server to Integration Server. If yo u do not select a truststore alias, the default truststore alias specified in the watt.security.trustStoreAlias property will be used.
Time Interval (seconds)
Specifies how often you want your server to check the mail server for incoming email messages (service requests).
Log out after each mail check
For use with IMAP and multi-threading only. Specifies whether you want your server to log out after checking for mail. If you select Yes, your server will log out a read-only thread to the IMAP mail server after checking for mail on that thread. The main read/write thread to the IMAP server will remain intact. You should select Yes if the IMAP server limits the number of connections that can remain logged in. If you select No, all read-only threads remain intact.
Security
Use this section to specify the user name under which services requested through this email port will run.
Run services as user
Specifies the user name under which requested services are to run if your clients do not pass user name and password information via the $user and $pass parameters on the Subject line in the email message.
Require authorization within message
Specifies whether the clients will pass their user names and passwords via the $user and $pass parameters in the Subject line of the email. When you enable this option, the server executes email requests under the user name the client supplies in the $user parameter. If you select No, the server executes email requests under the user name you specify in Run services as user. (If you set Require authorization within message to No, you must specify a user name in Run services as user.)
Use JSSE
Select Yes to use the JSSE library to create a port that supports TLS 1.3, TLS 1.2, or TLS 1.1.
Select No to use the Entrust IAIK library to create a port that supports only TLS 1.0 which is not secure.
Message Processing
Use this section to specify the way you want your server to process requests that it receives through the email client.
Global Service
Optional. Specifies the service that you want your server to execute for every email request. When you enable a global service, the server ignores the service parameter on the subject line of the email message.
Default Service
Optional. Specifies the service that you want your server to execute if the email message does not supply a valid service in the Subject line and the Global Service field is blank.
Send reply email with service output
Specifies whether you want the server to send any output that the service generates to the original sender in an email attachment. If you enable this option and the original email contained multiple attachments, the reply will contain an equal number of attachments.
Send reply email on error
Specifies whether you want the server to report errors to the original sender in the body portion of an email.
Delete valid messages (IMAP only)
Specifies whether you want the server to delete a valid email from the IMAP server once the it has successfully received the email. This setting helps prevent old emails from accumulating on the IMAP server, possibly affecting disk space and performance. The Integration Server always deletes emails on a POP server.
Delete invalid messages (IMAP only)
Specifies whether you want the server to delete invalid emails from the IMAP server. Invalid e-mails are those that reference services that cannot be invoked. For example, if the referenced service does not exist, the server will delete the email. If the service was invoked, but encountered errors, the server considers the associated e-mail to be valid. This setting helps prevent invalid emails from accumulating on the IMAP server, possibly affecting disk space and performance. The Integration Server always deletes emails on a POP server.
Multithreaded processing (IMAP only)
Specifies whether you want the server to use multiple threads for this port. This setting allows the port to handle multiple requests at once, thus avoiding a bottleneck.
Number of threads if multithreading turned on
Specifies the number of threads you want the server to use for this port. The default is 3.
Invoke service for each part of multipart message
Specifies whether the Integration Server invokes the service for each part of a multipart message or just once for the entire message. If you specify No, the entire email is passed to the appropriate content handler and then to the specified service for execution. If you specify Yes, the Integration Server treats each part of the message individually. That is, the Integration Server sends each part to the content handler and then to the specified service. If you specify Yes, do not include the email headers from the beginning of the message, because each section has its own headers that the content handler and/or the service already knows how to process. To exclude the email headers, specify No in the Include email headers when passing message to content handler field.
Include email headers when passing message to content handler
Specifies whether the Integration Server includes the email headers when passing an email message to the content handler. The email headers are typically found at the beginning of an email message. Specify Yes if you are processing a multi-part message as a single message. This ensures that the content handler and/or service can properly process the body of the email. Specify No if you are processing the different parts of an email individually (that is, if you specified Yes in the Invoke service for each part of multipart message field).
Email body contains URL encoded input parameters
Specifies how the IS treats input parameters it finds in email messages.With this value set to Yes, the IS considers a string such as ?one=1+two=2 to be a URL encoded input parameter. It then decodes this string into an IData object, puts it into the pipeline, and passes it to the service. With this value set to No, the IS treats the string as plain text and passes it to the appropriate content handler.
Save Changes
Updates the server with the new or changed port settings. Your changes take effect immediately.
Note:
Whenever you create or edit an email port, you must clear the browser cache. This applies only if you are using the OAuth authentication type.
