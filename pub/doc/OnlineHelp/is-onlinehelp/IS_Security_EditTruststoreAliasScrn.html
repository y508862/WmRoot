Edit Truststore Alias
Use the Edit Truststore Alias page to edit the properties of a specific truststore alias. A truststore contains the trusted root of the certificate or signing authority (CA). The truststore alias is a user-assigned identifier for the truststore, that can be easily referenced from within Integration Server.
Click Return to Keystore to return to the list of truststore aliases that are currently defined on the server.
Truststore Properties
Alias
Displays the user-specified identifier for the truststore. Clicking on the truststore alias displays its properties.
Description (optional)
Specifies a user-specified description of the alias.
Type
Specifies the file format of the truststore file. The default is JKS. Other truststore types can be made available by loading additional security providers, or by setting the watt.security.TrustStoreTypes server configuration property.
Provider
Specifies the provider that is used for the truststore type. The default is the provider shipped with the JVM (Oracle, IBM, or others). If your Hardware Security Module (HSM) device is not supported by the default provider, you can configure a different provider to support truststore file types.
Note:Integration Server only supports the JKS file format for truststores.
Location
Specifies the location of the truststore file on the server. This can be the full pathname, or a relative path in relation to the Integration Server.
Password/Re-type Password
Specifies the password for the saved keystore file associated with this alias. This password must have been defined at the time the truststore was created, using your keystore utility. Be sure that you have this password available when managing truststore aliases, as you cannot access it through the Integration Server.
Secondary Truststore
Alias of the truststore to be used as a secondary truststore. The secondary truststore acts an extension of the primary truststore (i.e., the truststore for which this alias is being created).
For example, suppose that you are creating a truststore alias named "CompanyTruststore" that contains all of your organization's private CA certificates. If you want to include the trusted CA certificates from the DEFAULT_JVM_TRUSTSTORE in the trust verification, set the DEFAULT_JVM_TRUSTSTORE as the secondary truststore. When preforming trust verification, Integration Server first checks the "CompanyTruststore" for an issuer. If there is no match, Integration Server checks the secondary truststore. In this example, by using a secondary truststore, you can separate out private certificates from public certificates but still include both in the trust verification.
