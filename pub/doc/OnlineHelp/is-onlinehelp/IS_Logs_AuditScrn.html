Service
Use the Service page to view the most recent Service Log (WMSERVICE_yyyymmdd_hhmmss.log) entries. The Service Log maintains an audit trail for all services that the server executes. Depending on the Audit property settings you specified for the service in Software AG Designer, the server can add records to the log at various times, such as when the service starts and again when it ends, or only when there is an error. For more information about collecting auditing information about services, see webMethods Service Development Help. For information about changing the display of this log, see webMethods Audit Logging Guide.
Note:
For security purposes, you should regularly check your audit logs for security breaches. In addition, you should back them up regularly in case the machine where the logs are stored becomes unavailable.
Service log columns
Note:
The Service page might not display all of the columns listed below. Click > Show columns to hide or display columns for the service log.
Date/Time
Date and time the audit record was written to the database or file system.
Audit time stamp
Date and time the entry was written to the log.
Service name
Name of the service that generated the audit event.
Status
Status of service execution. The status can be either STARTED, ENDED or FAILED.
Duration
How long the service ran (in milliseconds) before the error occurred.
Error Message
Optional error message, describing the type of error.
Server ID
IP address or host name of the server on which the Integration Server that generated the audit event is running. This is necessary information when Integration Servers are clustered and writing to a shared RDBMS. The ID can be DNSname:port or IPaddress:port.
Note:
The port is always the Integration Server’s primary port, even if the event occurred on a different (non‑primary) Integration Server port.
User ID
Identity of the user that invoked the service that generated the audit event.
Resubmittable
Whether or not webMethods Monitor can resubmit the service. A service is resubmittable only if it is a top-level service and the pipeline was included.
Client ID
IP address of the host from which the request originated.
Pipeline flag
Whether the service log includes a copy of the input pipeline.
System data flag
Whether system data is available.
Audited parent ID
Context ID of the next higher service in the call stack that was audited.
Client application
Client application accessing the server. If this is an OAuth related request and the request has an access_token, this holds the client_id to which the token was issued.
Root context
Context information that provides a consistent, interpretable audit trail of activity that the webMethods Monitor can use to relate entries within the log.
Parent context
Context information that provides a consistent, interpretable audit trail of activity that the webMethods Monitor can use to relate entries within the log
Current context
Context information that provides a consistent, interpretable audit trail of activity that the webMethods Monitor can use to relate entries within the log
Custom context
Arbitrary value that users can attach to a transaction. All services that get audited for that transaction will have the same custom context. Alternatively, a custom context is passed in a messaging envelope.
