Security
Use the Security page to view the contents of the most recent security audit log (WMSECURITY_yyyymmdd_hhmmss.log). The security audit log maintains an audit trail for all security actions that the server executes. Depending on the settings you specify for Security Configuration and Security Actions on the Settings > Logging screen, the server adds records to the log at various times, such as when security auditing was enabled or when auditing configuration was modified.
Note:
For security purposes, you should regularly check your audit logs for security breaches. In addition, you should back them up regularly in case the machine where the logs are stored becomes unavailable.
Security log columns
Note:
The Security page might not display all of the columns listed below. Click > Show columns to hide or display columns for the security log.
Date/Time
Date and time the audit record was written to the database or file system.
Audit time stamp
Date and time the entry was written to the log.
Message
Provides information about the security action.
Server ID
IP address or host name of the server on which the Integration Server that generated the audit event is running. This is necessary information when Integration Servers are clustered and writing to a shared RDBMS. The ID can be DNSname:port or IPaddress:port.
Note:
The port is always the Integration Server’s primary port, even if the event occurred on a different (non‑primary) Integration Server port.
Client ID
IP address of the host from which the request originated.
When watt.server.securityLog.ignoreXForwardedForHeader is set to false, Integration Server obtains the originating client IP address from the X-Forwarded-For request header and uses that client IP address as the Client Id in the security log and security log message. When watt.server.securityLog.ignoreXForwardedForHeader is set to true, Integration Server ignores the X-Forwarded-For request header and uses the IP address of the proxy server as the client Id.
User ID
Identity of the Integration Server user performing this security action.
When watt.server.securityLog.logAnonymousRequests is set to true, the security logger will write anonymous authentication requests to the security log. In a log entry, the User Id for an anonymous request will be "local/default".
Security Event Type
A specific area in Integration Server for which a security action was performed.
Result
The result of the request for which security action was logged, either SUCCESS or FAILURE.
Service name
Name of the service that generated the audit event.
Client application
Client application accessing the server. This could be an OAuth client application.
Request size
Size of the request in bytes. This is for OAuth events.
Root context
Context information that provides a consistent, interpretable audit trail of activity that the webMethods Monitor can use to relate entries within the log.
Parent context
Context information that provides a consistent, interpretable audit trail of activity that the webMethods Monitor can use to relate entries within the log
Current context
Context information that provides a consistent, interpretable audit trail of activity that the webMethods Monitor can use to relate entries within the log
